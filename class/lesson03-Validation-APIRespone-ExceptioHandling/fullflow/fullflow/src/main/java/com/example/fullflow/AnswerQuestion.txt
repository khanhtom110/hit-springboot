1. @NotNull, @NotEmpty, @NotBlank khác nhau thế nào? Khi nào dùng cái nào?
WHAT
-> 3 anh này đều kiểm tra null
-> @NotNull sẽ kiểm tra null k kiểm tra size và trim
-> @NotEmpty sẽ kiểm tra null, size và không kiểm tra trim
-> @NotBlank kiểm tra null, size và trim
WHEN
-> String hay dung @NotBlank
-> Danh sách hay dung @NotEmpty (do size có thể =0)
-> Só hay dung @NotNull (số 0 só size -chỉ có số ||null, còn về khoảng trắng thì đã có Interger.parseInt())

2. Tại sao validate @RequestBody dùng @Valid nhưng validate @RequestParam lại cần @Validated ở class level?
WHAT
-> @RequestBody là truyền dữ liệu trong body của HTTP Request, dữ liệu dạng JSON.
   Khi dùng @RequestBody, Spring sử dụng HttpMessageConverter -> biến chuỗi JSON thành một Object Java
   Mà @Valid để kiểm tra trong lúc convert JSON -> Object
-> @RequestParam là truyền dữ liệu trên thanh url, dữ liệu dạng key value, thường để lọc
   Các dữ liệu trên url là dl ng thủy
HOW
-> Khi gọi đến các tham số -> sẽ tạo ra các vỏ (proxy) -> ktra các điều kiện như @Min(1) -> false thì trả về exception ConstraintViolationException
TẠI SAO PHẢI ĐẶT Ở ĐẦU CLASS
-> Tạo proxy này tốn bộ nhớ -> Chỉ cần mới dùng

5. @ControllerAdvice hoạt động như thế nào? Tại sao giúp code sạch hơn try-catch?
WHAT
-> @ControllerAdvice để bắt lỗi của controller 
-> Khi lỗi ở controller, thay vì văng lỗi ra ngoài thì nó văng vào Advice này
Tại sao giúp code sạch hơn try-catch?
-> Đỡ dài, đỡ sót các lỗi không bắt
-> Ngoài ra, các tầng như sercurity thì không thề ném exception vào @ControllerAdvice
